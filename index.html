MomEase – Maternal Wellness App with AI Companion

Overview

MomEase is a cross-platform mobile application designed to support maternal mental health by coupling self-assessments, secure journals, and structured analytics with an AI companion. The project was built using a clean decoupling model—a native-feeling mobile client communicating with a secure backend server via standardized REST endpoints.

Problem

Mothers experiencing symptoms of postpartum depression (Baby Blues) need a secure, reliable space to log and analyze their emotional state. Most available apps lack:

  • Secure API endpoints to ensure user privacy for highly personal data.
  • Relational schemas capable of mapping user entries, self-assessments, and recommendations efficiently.
  • Low-latency data synchronization between local app state and remote storage.

Solution

We engineered a secure full-stack mobile system featuring:

  • A relational MySQL schema designed to store user data, journals, and self-assessment scores with high integrity.
  • A robust Laravel REST API backend delivering secured endpoints for mobile requests.
  • A native-performing Flutter application for smooth onboarding and interactive tracking charts.

My Role

Backend Developer & Database Architect

  • Designed and normalized the MySQL database schema to handle secure journal entries and test scores.
  • Developed and documented Laravel REST APIs for authentication, journal CRUD operations, and assessment submissions.
  • Implemented token-based session verification to protect user data privacy across requests.
  • Integrated backend API endpoints with the Flutter client, ensuring proper JSON payload contracts.

Tech Stack

Frontend Flutter (Dart)
Backend Laravel (PHP)
Database MySQL
API Design REST API

Challenges

  • Designing a normalized database schema that supports multiple features (journals, self-tests, article catalog) without excessive joins.
  • Establishing secure token authentication and handling cross-origin requests cleanly between Flutter and Laravel.
  • Structuring API responses to minimize payload size and optimize rendering speed on the mobile client.

Impact

  • Delivered a robust, high-performance back-end system with structured, normalized MySQL schemas.
  • Achieved seamless integration between the Flutter mobile client and the Laravel backend API.
  • Adopted structured system design principles, resulting in a clean separation of concerns and maintainable code.

DATA PIPELINE

Vaccine Demand Prediction: Distributed ML Analytics Pipeline

Overview

This project is a distributed data processing and predictive analytics pipeline engineered to forecast weekly COVID-19 vaccine needs across EU/EEA countries. The pipeline leverages Apache Spark to clean, transform, and model over 800,000 vaccination records from the European Centre for Disease Prevention and Control (ECDC). It transitions static statistical analysis into a scalable data engineering workflow.

Problem

Vaccine distribution logistics face severe supply-demand imbalances due to fluctuating regional trends. Building an analytics pipeline for this problem requires solving:

  • Inability of single-node architectures (like Pandas/Scikit-learn) to scale as datasets grow.
  • Complex feature engineering requirements for time-series lag variables across multiple geographical subsets.
  • Handling missing data and data anomalies across heterogenous reporting standards.

Solution

We engineered a high-throughput distributed pipeline featuring:

  • PySpark data preprocessing scripts that normalize, clean, and clean historical vaccination logs in parallel.
  • Distributed lag-feature generation algorithms to compute time-series steps without data loss.
  • Distributed Random Forest regression models trained on country-partitioned data clusters.

My Role

Backend & Data Pipeline Developer

  • Programmed the data ingestion and transformation ETL stages using PySpark.
  • Engineered distributed time-series lag features and rolling averages on Spark DataFrames.
  • Configured distributed Random Forest regression pipelines utilizing Spark MLlib.
  • Implemented validation metric pipelines (RMSE, MAE, R²) to benchmark model performance across country groups.

Tech Stack

Engine Apache Spark
Language Python (PySpark)
Algorithms Random Forest
Dataset Size 800K+ Records

Challenges

  • Optimizing time-series windows in PySpark to calculate historical lags without causing massive data-shuffling operations across cluster nodes.
  • Handling inconsistent reporting intervals and data gaps across 30 EU/EEA countries using robust imputation algorithms.
  • Preventing model overfitting while maintaining high generalization capabilities across countries with highly varying population sizes.

Impact

  • Formulated a high-performance big data analytics pipeline capable of distributed scaling.
  • Achieved predictive model reliability (R² ≈ 0.83) on large-scale vaccination subsets.
  • Shifted from single-node analytics to production-grade distributed data pipelines.

What I'd Improve Next

  • Integrate real-time stream processing using Spark Structured Streaming for live database updates.
  • Deploy the prediction models inside Docker containers as microservices called via REST endpoints.
  • Benchmark performance against GBTs (Gradient Boosted Trees) and LSTMs implemented on distributed clusters.

SYSTEM INTEGRATION

Thermal Camera & YOLOv11 Automated Detection System

Overview

This project is a real-time nocturnal rat detection system combining thermal camera imagery, OpenCV image processing, and optimized YOLOv11 models, deployed to solve crop-destruction pest problems in Desa Sengka. The system integrates custom python detection logic with automatic physical hardware controls to activate physical ultrasonic deterrents when a rat is located.

Problem

Farmers lose significant crop yields to nocturnal rodent attacks. Traditional solutions fail due to:

  • Difficulty tracking pests in complete darkness without active illumination.
  • Inability to trigger real-time physical deterrents automatically upon pest location.
  • Lack of low-power, edge-deployable software architectures suitable for farm areas.

Solution

We engineered a closed-loop automated system featuring:

  • Custom preprocessing python scripts that convert and scale multi-source thermal datasets.
  • An augmented image processing pipeline using OpenCV to handle textureless thermal signatures.
  • A low-latency integration script that triggers serial signals to hardware microcontrollers upon high-confidence detections.

My Role

Software & ML Engineer • Lead Author

  • Unified raw thermal datasets, writing scripts to convert heterogeneous segmentation masks into normalized YOLO coordinates.
  • Developed custom image augmentation algorithms in OpenCV to enhance low-contrast thermal boundaries.
  • Trained and optimized the YOLOv11 neural network topology to minimize inference latency.
  • Programmed the hardware response scripts that send signal codes to physical ultrasonic modules.
  • Authored the peer-reviewed research paper detailing the system's software-hardware architecture.

Tech Stack

Libraries YOLOv11, OpenCV
Integration Hardware GPIO Response
Language Python

Challenges

  • Overcoming the absence of RGB texture and color features in thermal feeds by applying custom intensity histogram slicing.
  • Handling different image sizes and annotations (masks vs. bounding boxes) to create a clean, uniform training dataset.
  • Minimizing inference delay on edge devices to prevent the rat from leaving the target area before the ultrasonic signal triggers.

Impact

The integrated system achieved an object detection mAP@50 of 0.99+, with precision of 0.989 and recall of 0.97+. The automated Python control loop successfully established zero-loss signal handshakes to trigger the ultrasonic deterrent, offering farmers a hands-free, early-warning crop security system.

What I'd Improve Next

  • Port the entire detection and control codebase to run locally on a Raspberry Pi 5 with an attached thermal camera.
  • Develop an asynchronous MQTT logging system to publish detection counts to a central dashboard.
  • Implement target-tracking algorithms (e.g., DeepSORT) to prevent multiple triggers for the same pest.

Order Here – Smart Food Ordering Ecosystem

Overview

Order Here is a smart food ordering ecosystem developed for Universitas Ciputra Makassar to modernize the campus dining experience through artificial intelligence, IoT, and web technologies.

Instead of creating a standalone recommendation model, our team designed a complete ecosystem consisting of a web-based ordering platform, MoodBites mobile application, IoT-powered NFC authentication kiosk, cloud backend, and an AI recommendation microservice.

My primary contribution focused on designing and developing the machine learning recommendation engine that generates personalized food recommendations based on users' taste preferences, favorite foods, and weighted flavor priorities.

Problem

Campus cafeterias typically provide static menus with no personalization, making it difficult for students to discover meals that match their preferences while maintaining a smooth ordering experience.

  • Food recommendations were generic rather than personalized.
  • First-time users had no recommendation history.
  • Ordering process required manual identification and payment.
  • Existing systems lacked integration between ordering, authentication, and recommendation services.

Solution

We built a complete smart food ordering ecosystem consisting of multiple interconnected services.

  • FastAPI recommendation microservice deployed independently using Docker.
  • Web-based ordering platform for campus cafeteria operations.
  • MoodBites mobile application allowing users to submit their taste preferences.
  • IoT kiosk utilizing Raspberry Pi and PN532 NFC reader for student authentication.
  • Cloud deployment using VPS and Supabase for centralized data management.

System Architecture

The recommendation engine was designed as an independent microservice integrated with the larger Order Here ecosystem.

DATA SCIENCE COMPETITION

COPPA Risk Classifier – Automated ML Pipeline

Overview

A predictive classification system developed for the FIND IT 2025 Data Analytics Competition, designed to automatically flag mobile applications that risk violating the Children's Online Privacy Protection Act (COPPA). The engineering focus was on building a robust, automated ML pipeline — from raw app metadata to a production-ready classifier — using TPOT AutoML to eliminate manual hyperparameter search.

Problem

Manually reviewing apps for COPPA compliance at scale is infeasible. The engineering challenges were:

  • App metadata contains hundreds of heterogeneous, noisy, and correlated features requiring careful selection.
  • Manual model and preprocessor selection across this feature space is prohibitively slow.
  • Class imbalance between COPPA-compliant and non-compliant apps degrades naive classifiers.

Solution

  • Conducted structured exploratory data analysis (EDA) to identify high-signal feature subsets from app metadata.
  • Engineered 17 domain-informed predictive features including child-rating indicators and age-restriction flags.
  • Configured TPOT AutoML to run evolutionary search across scikit-learn preprocessing and classifier combinations.
  • Benchmarked and selected the best generated pipeline based on F1-score against a held-out validation set.

My Role

ML Pipeline Engineer

  • Designed and implemented the full feature engineering pipeline in Python/Pandas.
  • Configured TPOT's evolutionary algorithm parameters to balance search depth and runtime budget.
  • Evaluated and compared dozens of AutoML-generated pipelines using precision, recall, and F1 metrics.
  • Documented pipeline decisions, model interpretations, and competition findings.

Tech Stack

Language Python
AutoML TPOT
ML Library Scikit-learn
Task Binary Classification

Challenges

  • Engineering discriminative features from raw app store metadata with high dimensionality and noise.
  • Configuring TPOT's search budget to explore enough pipeline candidates without exceeding runtime limits.
  • Interpreting opaque AutoML-generated pipelines to validate that predictions are driven by meaningful signals.

Impact

  • Delivered a fully automated COPPA risk classifier pipeline validated on a held-out competition dataset.
  • Eliminated weeks of manual hyperparameter tuning through systematic AutoML pipeline search.
  • Demonstrated an end-to-end ML engineering workflow — from raw data ingestion to submission-ready predictions.

What I'd Improve Next

  • Add SHAP explainability layer to audit which features drive individual predictions.
  • Expose the trained pipeline as a FastAPI REST endpoint for on-demand app compliance scoring.
  • Expand the training dataset with additional app store metadata and recent Play Store entries.
